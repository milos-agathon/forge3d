# forge3d 3D Map Rendering Quality — Blender-Outmatch Implementation Plan

**Date:** 2026-07-02

**P0/P1 completion status:** All BOP-P0 and BOP-P1 tasks are implemented and captured for the `1.30.0` package/PyPI release on 2026-07-04; residual polish is listed in the 2026-07-04 implementation audit section. P2/P3 closure is captured for the `1.31.0` package/PyPI release on 2026-07-09.

**Scope:** Diagnosis and phased implementation plan for forge3d's 3D map rendering stack: Rust rendering architecture under `src/`, WGSL shaders under `src/shaders/`, the Python public API under `python/forge3d/`, and rendering examples under `examples/`. This is a planning document only; no source changes accompany it. Evidence is cited as `path:line` against the repository state at commit `00b5803` (branch `codex/g001-p1-1-recipe-manifest`).

**Executive summary:** forge3d already contains a genuinely strong, regression-tested single-tile terrain renderer — `terrain_pbr_pom.wgsl` with layered materials, POM, IBL, cascade shadows including moment variants, heightfield ray-marched AO and sun visibility, volumetrics, and a real HDR offline accumulation pipeline with AOVs, denoising, and SSIM-gated golden tests. Almost everything else the "3D map" promise needs exists somewhere in the repo but is disconnected: the flagship `MapScene` API renders CPU placeholder rectangles instead of invoking the GPU renderer; the deterministic Rust label engine, AA vector pipelines, and OIT never feed offscreen map output; CRS/geotransform correctness is performed by rasterio/pyproj inside example scripts and severed before the GPU; the richest subsystems (wavefront/ReSTIR path tracer, 3D Tiles traversal, COPC point-cloud decode, clipmap LOD, GPU culling) have zero Python bindings or are dead code; and several public Python classes (`PathTracer`, `materials.py`, MapScene building/point-cloud layers) are stubs that silently return synthetic output. The highest-leverage plan is therefore not new rendering research but surgical integration: wire the existing GPU terrain, label, vector, and offline-accumulation machinery into `MapScene`, move geospatial alignment into the library, close a small number of true quality gaps (glyph rendering, line joins, per-texel terrain material maps, building rendering, sky-model fidelity), and lock everything behind an expanded golden-image gate. That sequence converts forge3d from "impressive demos plus a placeholder API" into a reproducible, geospatially correct, cartographic rendering engine that Blender cannot match in its domain.

## What "outmatch Blender for 3D map rendering quality" means

The target is **not** generic feature parity with Blender or beating Cycles at open-ended photorealism. It means that for the specific job of producing high-end, reproducible 3D map renders — DEM + imagery + vector overlays + labels + buildings + map furniture — forge3d must be strictly better on these axes:

1. **Geospatial correctness by default.** CRS-aware ingest, reprojection, and sub-pixel raster/vector/label alignment as library behavior, with alignment diagnostics. Blender has no native CRS concept; georeferenced workflows rely on third-party add-ons (e.g., BlenderGIS) and lose datum/projection metadata *(general background knowledge — Blender's official manual was not reachable for verification during this audit; claims about Blender are deliberately conservative)*.
2. **Cartographic output as a first-class render primitive.** Deterministic, declutter-aware label placement with halos and leader lines, AA vector strokes with real joins, geodesically correct scale bars, legends, north arrows, graticules — rendered by the engine, not composited by hand. Blender has no label placement or map-furniture system *(general background knowledge)*.
3. **Reproducibility as a contract.** One Python script or serialized scene recipe → byte-stable placement decisions, seeded sampling, deterministic camera paths, and SSIM-gated golden tests in CI. Blender scenes are reproducible in principle but there is no built-in regression-gating workflow for render output *(general background knowledge)*.
4. **Time-to-premium-map.** From `DEM file + style` to a print-quality, georeferenced render in one short script using documented public API — no 70-flag CLI archaeology, no add-on assembly.
5. **Sufficient photometric quality.** Physically plausible sun/sky, IBL, soft shadows, AO/volumetrics, HDR accumulation + denoise + EXR/16-bit output — good enough that the map render does not look like a "GIS screenshot." forge3d does not need to exceed Cycles at global illumination; it needs to remove any visible quality reason to export a map to Blender.

Where Blender is acknowledged stronger and this plan does **not** compete: open-ended path-traced photorealism, general 3D content authoring/modeling, animation tooling, and the add-on ecosystem.

## Architecture reality (root-cause findings)

These five findings explain most axis-level gaps and drive the plan's shape:

1. **Three disjoint render stacks.** (a) The real GPU terrain renderer (`TerrainRenderer`/`TerrainScene`, mega-shader `src/shaders/terrain_pbr_pom.wgsl`, ~4,180 lines, assembled in `src/terrain/renderer/pipeline_cache.rs:55-118`) reached via `render_terrain_pbr_pom` (`src/terrain/renderer/py_api.rs:69`) and the interactive viewer (`src/terrain/renderer/viewer.rs:4,175`). (b) A separate `src/scene/` stack that owns SSAO/SSGI/SSR. (c) CPU/PIL compositors: `MapScene`'s fallback renderer (`python/forge3d/_map_scene_render.py:80-163` — hash-seeded gradients, labels as solid rectangles) and the OSM city/daycycle/solar examples (`examples/osm_city_daycycle.py`, `examples/rotterdam_solar_potential_shadow_study.py` import PIL/pyproj, never the GPU renderer).
2. **The flagship API renders placeholders.** `MapScene.render()` uses the native GPU path only when terrain is a 2D `.npy` and overlays are `.png` (`python/forge3d/map_scene.py:125-151, 178-219`); GeoTIFF DEMs, vectors, labels, buildings, and point clouds fall back to the CPU placeholder. Building/point-cloud/3D-Tiles layers default to `support_level="underdeveloped"` and emit placeholder diagnostics (`map_scene.py:1492-1656`).
3. **Geospatial correctness lives in examples, not the library.** `crs.py` always uses pyproj because the native `proj` feature is compiled off (`python/forge3d/crs.py:114-149`; `src/geo/reproject.rs:131-137,162`; `pyproject.toml:109` enables only `extension-module, weighted-oit, enable-tbn, enable-gpu-instancing, copc_laz, gis-remote`). Flagship viewers reproject DEMs with rasterio themselves (`examples/swiss_terrain_landcover_viewer.py:198-233`), and the native viewer loader discards nodata, CRS, and geotransform entirely (`src/terrain/renderer/viewer.rs:49-92`).
4. **Rich Rust capability with zero Python exposure or dead wiring.** No pyo3 bindings: the wavefront/ReSTIR path tracer (`src/path_tracing/wavefront/` — `WavefrontScheduler` referenced nowhere outside its module). Render integration still missing for 3D Tiles/COPC despite native decode/traversal exposure. Previously suspected `culling_compute.wgsl` and `extrusion.wgsl` are live includes; prior `clipmap_terrain.wgsl`, `terrain_velocity.wgsl`, and generic `src/core/virtual_texture/` stub were deleted in BOP-P2-08. `src/terrain/page_table/` and `src/terrain/stream/` are live through TerrainSpike/COG infrastructure, not dead cleanup targets. Clipmap GPU LOD remains a BOP-P2-02 integration concern rather than BOP-P2-08 dead-code cleanup.
5. **Stub APIs that misrepresent capability.** `forge3d.path_tracing.PathTracer.render_rgba` synthesizes seeded noise and ignores geometry (`python/forge3d/path_tracing.py:167-275`); `render_aovs` fabricates procedural AOVs (`path_tracing.py:420-458`). `materials.py` is a 28-line test container; `textures.py:60-63` returns `data=None` from disk loads. `RestirDI` raises at render because the native functions don't exist (`python/forge3d/lighting.py:220,350`). `SkySettings` advertises "Rayleigh and Mie scattering" (`python/forge3d/terrain_params.py:1197-1206`) but `sky.wgsl` is an approximate analytic model with hand-tuned coefficients (`src/shaders/sky.wgsl:83-84,220-240`). Conversely, stale notes claim VSM/EVSM/MSM are unimplemented (`terrain_params.py:55,150-164`) while the shader actually samples moment maps (`src/shaders/terrain_pbr_pom.wgsl:926-1153`).

### BOP-P0-01 — Wire `MapScene.render()` to the real GPU terrain renderer with GeoTIFF ingestion

## Dependency policy

This plan adds **one** new dependency, in P3: `rustybuzz` (+ `unicode-bidi`) for text shaping (BOP-P3-03), justified because no existing dependency shapes OpenType text and the capability is impossible without it. Everything else deliberately reuses what's present: lyon (stroke/fill tessellation), image crate (16-bit PNG), `gltf` crate (materials it already parses), PIL+scipy Python-side (SDF atlas baking), rasterio/pyproj (alignment), reqwest/tokio (already shipped via `gis-remote`), Hosek-Wilkie coefficient tables (data, not code).
